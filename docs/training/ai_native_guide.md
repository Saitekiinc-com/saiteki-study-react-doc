# AIネイティブ・エンジニアリング指針

AI（LLM）の進化により、エンジニアに求められるスキルセットは大きく変化しています。
本研修では、AIを「禁止」するのではなく、**「最強のパートナーとして使いこなし、AIよりも上位の視座を持つ」**ことを目指します。

## 1. マインドセットの転換

### ❌ Old: Code Writer (コードを書く人)
*   「文法を暗記していること」が価値。
*   0から100まで自分でタイピングすることにこだわる。
*   AIを使うことを「ズル」だと思っている。

### ⭕️ New: Code Reviewer & Architect (設計し、判断する人)
*   **「何を作るか（要件）」と「なぜそうするか（設計）」を言語化できること**が価値。
*   AIにコードを書かせ、人間はそれを**レビュー（検証・修正）**する。
*   AIは「優秀だが、たまに嘘をつく部下」。その嘘を見抜くための基礎力が不可欠。

## 2. AI時代に求められるスキル

コーディングの自動化が進むからこそ、以下のスキルがより重要になります。

### ① 基礎知識と原理原則 (Lv.1 Foundation)
AIが書いたコードが「正しいか」「セキュリティ的に安全か」「パフォーマンスは問題ないか」を判断するためには、基礎知識が必要です。
*   **なぜ学ぶのか**: AIのハルシネーション（もっともらしい嘘）に騙されないため。

### ② 言語化能力 (Prompt Engineering)
AIは「指示されたこと」しかやりません。曖昧な指示は、バグの温床になります。
*   **要件定義力**: 「こういう機能が欲しい」ではなく「入力はA、処理はB、出力はC。例外時はD」と具体的に定義する力。
*   **コンテキスト管理**: プロジェクトの前提条件や制約事項をAIに正しく伝える力。

### ③ デバッグとトラブルシューティング (Lv.2 Application)
AIは完璧ではありません。動かないコードが出力されたとき、エラーログを読み、原因を特定し、AIに修正指示を出す（あるいは自分で直す）能力が必要です。

## 3. 研修におけるAI活用ガイドライン

本研修では、以下のルールでAIを活用してください。

### ✅ 推奨される使い方
1.  **「解説」を求める**:
    *   「このコードの `useEffect` はなぜ必要なの？わかりやすく解説して」
    *   「このエラーログはどういう意味？原因の仮説を3つ挙げて」
2.  **「レビュー」を求める**:
    *   「自分で書いたこのコード、もっと可読性を上げるにはどうしたらいい？」
    *   「セキュリティ的な脆弱性はない？」
3.  **「テストデータ・テストケース」の生成**:
    *   「この関数の単体テストのパターンを網羅的にリストアップして」

### 🚫 禁止・非推奨な使い方
1.  **思考停止のコピペ**:
    *   出力されたコードを**1行1行理解せずに**貼り付けること。
    *   → **対策**: コピペする前に、必ず「なぜこのコードで動くのか」を自分（またはAI）に説明させること。
2.  **要件の丸投げ**:
    *   「いい感じのTodoアプリ作って」
    *   → **対策**: 画面構成、機能要件、データ構造を自分で設計してから、実装を依頼すること。

## まとめ
**「AIに使われるな、AIを使いこなせ。」**
コードを書く時間は減りますが、考える時間は増えます。より本質的なエンジニアリング（課題解決）に集中しましょう。
