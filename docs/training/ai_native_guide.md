# AIネイティブ・エンジニアリング指針

AI（LLM）の進化により、エンジニアに求められるスキルセットは大きく変化しています。
本研修では、AIを「禁止」するのではなく、<strong>「最強のパートナーとして使いこなし、AIよりも上位の視座を持つ」</strong>ことを目指します。

## 1. マインドセットの転換

なぜマインドセットを変える必要があるのでしょうか？
AIの登場により、「コードを書くこと」自体の市場価値は低下傾向にあります。一方で、「どのようなシステムを作るか（要件定義）」や「AIが書いたコードが正しいか（品質保証）」という、<strong>上流と下流の工程</strong>の重要性が相対的に高まっているからです。

これからのエンジニアは、コーディング作業そのものではなく、<strong>「AIという強力なエンジンを使って、どれだけ早く・高品質に価値を届けられるか」</strong>で評価されます。

### ❌ Old: Code Writer (コードを書く人)
*   「文法を暗記していること」が価値。
*   0から100まで自分でタイピングすることにこだわる。
*   AIを使うことを「ズル」だと思っている。

### ⭕️ New: Code Reviewer & Architect (設計し、判断する人)
*   <strong>「何を作るか（要件）」と「なぜそうするか（設計）」を言語化できること</strong>が価値。
*   AIにコードを書かせ、人間はそれを<strong>レビュー（検証・修正）</strong>する。
*   AIは「優秀だが、たまに嘘をつく部下」。その嘘を見抜くための基礎力が不可欠。

## 2. AI時代に求められるスキル

コーディングの自動化が進むからこそ、以下のスキルがより重要になります。

### ① 基礎知識と原理原則 (Lv.1 Foundation)
AIが書いたコードをレビューする時だけでなく、<strong>「AIに指示を出す（設計・計画）」段階</strong>でも基礎知識が必要です。
正しい知識がないと、AIに適切な指示が出せず、意図しない実装になってしまいます。
*   <strong>計画時</strong>: 実現可能なアーキテクチャやライブラリを選定するため。
*   <strong>レビュー時</strong>: AIのハルシネーション（もっともらしい嘘）や、セキュリティリスクを見抜くため。
*   <strong>なぜ学ぶのか</strong>: AIのハルシネーション（もっともらしい嘘）に騙されないため。

### ② 言語化能力 (Prompt Engineering)
AIは「指示されたこと」しかやりません。曖昧な指示は、バグの温床になります。
*   <strong>要件定義力</strong>: 「こういう機能が欲しい」ではなく「入力はA、処理はB、出力はC。例外時はD」と具体的に定義する力。
*   <strong>コンテキストエンジニアリング</strong>: プロジェクトの前提条件（使用技術、ディレクトリ構成、命名規則など）や制約事項をAIに正しく伝え、文脈を制御する力。

### ③ デバッグとトラブルシューティング (Lv.2 Application)
AI開発ツール（AntigravityやCursorなど）を使えば、エラーの修正案も自動で提示してくれます。しかし、<strong>「その修正が本当に正しいのか」</strong>を判断するのは人間です。
ツールは「エラーを消すこと」を優先し、時には既存のロジックを壊す修正を提案することもあります。
*   <strong>根本原因の理解</strong>: 表面的なエラー修正ではなく、なぜそのバグが起きたのかを理解し、再発を防ぐ力。

### ④ 多角的な検証能力 (Multi-perspective Verification)
コードが「動く」だけでは不十分です。AIが見落としがちな「非機能要件」を人間が補完する必要があります。
*   <strong>セキュリティ</strong>: 脆弱性（XSS, SQLiなど）がないか。
*   <strong>パフォーマンス</strong>: 無駄な再レンダリングやN+1問題がないか。
*   <strong>ユーザー体験 (UX)</strong>: エラー時の表示や、操作性は適切か。
*   <strong>アクセシビリティ</strong>: スクリーンリーダー対応などはできているか。

## 3. 研修におけるAI活用ガイドライン

本研修では、以下のルールでAIを活用してください。

### ✅ 推奨される使い方
1.  <strong>「解説」を求める</strong>:
    *   「このコードの `useEffect` はなぜ必要なの？わかりやすく解説して」
    *   「このエラーログはどういう意味？原因の仮説を3つ挙げて」
2.  <strong>「レビュー」を求める</strong>:
    *   「自分で書いたこのコード、もっと可読性を上げるにはどうしたらいい？」
    *   「セキュリティ的な脆弱性はない？」
3.  <strong>「テストデータ・テストケース」の生成</strong>:
    *   「この関数の単体テストのパターンを網羅的にリストアップして」

### 🚫 禁止・非推奨な使い方
1.  <strong>思考停止のコピペ</strong>:
    *   出力されたコードを<strong>1行1行理解せずに</strong>貼り付けること。
    *   → <strong>対策</strong>: コピペする前に、必ず「なぜこのコードで動くのか」を自分（またはAI）に説明させること。
2.  <strong>要件の丸投げ</strong>:
    *   「いい感じのTodoアプリ作って」
    *   → <strong>対策</strong>: 画面構成、機能要件、データ構造を自分で設計してから、実装を依頼すること。

## まとめ
<strong>「AIに使われるな、AIを使いこなせ。」</strong>
コードを書く時間は減りますが、考える時間は増えます。より本質的なエンジニアリング（課題解決）に集中しましょう。
